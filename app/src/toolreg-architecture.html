Tool Registry — Architecture & Execution Flow

How toolreg-router, toolreg-async-cb, and toolreg-scheduler cooperate to broker every data-source call — sync DB queries, sync API calls, and async API patterns (webhook + polling) — shown end-to-end with the DDaaS context.

Router (gRPC + REST) Async callback receiver Scheduler (polling) Tool implementation / pool Vault External system / source Audit / state store

Execution paths

Sync path (most common)

1
Caller invokes execute. ADK agent calls execute(tool_id, params) over gRPC or MCP.
2
Router resolves & gates. Identity check, resolves tool_id → SourceTool, checks Redis-backed quotas.
3
Pool acquire. JDBC pool (UCP/DBCP2/Tomcat/Hikari) gives a connection. Vault is touched only at pool init, not per call.
4
Execute under guardrails. Statement timeout, row cap, byte cap; streamed back when large.
5
Audit + circuit breaker. Every call written to the audit store; Resilience4j updates source health.

Async path · Mode A (webhook)

1
Submit. Caller invokes executeAsync; router calls upstream, gets request_id.
2
Trace. Router persists async_trace {trace_id, request_id, status=pending}.
3
Webhook arrives. Upstream POSTs /callbacks/{trace_id} → toolreg-async-cb validates signature, updates trace, signals waiter.
4
Caller collects. Agent GET /v1/traces/{trace_id} or subscribes via SSE.

Async path · Mode B (polling)

1
Submit. Same as above — but upstream has no webhook.
2
Schedule. Router registers a Quartz job in toolreg-scheduler with exponential backoff.
3
Poll. Scheduler polls the upstream status endpoint until complete / failed / TTL exceeded.
4
Resolve trace. Same retrieval path as Mode A.

Service responsibilities

Router toolreg-router

  • Single entry point — gRPC + REST + optional MCP for execution.
  • Holds JDBC pools in-process; memory-heavy, scales horizontally.
  • Owns identity, quotas, rate limits, timeouts, circuit breakers.
  • Emits the audit record for every call.

Webhook toolreg-async-cb

  • Public-ish endpoint /callbacks/{trace_id}.
  • Signature verification per upstream.
  • Updates async_trace and notifies subscribers.
  • Stateless, separate scaling from router.

Scheduler toolreg-scheduler

  • Quartz cluster for resilient polling.
  • Backoff policy per upstream; TTL on traces.
  • No driver pools — purely orchestration.

DDaaS link

Tool Registry reads tool_definition from DDaaS at boot and on cache invalidation. Tool Registry only executes; DDaaS owns meaning.

What flows between Tool Registry and DDaaS

  • DDaaS → Tool Registry (during registration / profiling) — testConnection, describeSchema, sampleRows. Tool Registry is the only component opening source connections.
  • DDaaS → Tool Registry (at boot) — Tool Registry caches tool_definition records; invalidated by ddaas.dictionary.updated Kafka events.
  • Tool Registry → DDaaS (operational) — degraded-source events surface in the DDaaS UI so stewards can see which Tools are unhealthy.

Both services share a small internal platform-contracts Maven artifact (proto + generated Java stubs) so the contract is versioned and not duplicated.
