2.Executive Summary

Project O2A turns recorded human SOP execution into autonomous, data-aware agents. The Ingestion Pipeline emits a SKILL.md per SOP. Google ADK runs those agents. Between them sit the two components owned by this design: Data Dictionary as a Service (DDaaS) — the catalog that makes every SKILL.md data-aware — and the Tool Registry — the connection-pooled execution layer the running agents call.

This HLD covers both components end-to-end: functional design, event-driven pipeline, data model, APIs (REST + MCP), per-source connectors, sync/async API tools, governance, observability, deployment, and capacity. It also resolves the open question on JDBC pooling by introducing a pluggable ConnectionPoolProvider SPI, so the platform is not coupled to HikariCP and can run on Oracle UCP, Apache DBCP2, or Tomcat JDBC Pool depending on enterprise approval.

3.Scope & Assumptions

3.1 In scope Owned

  • DDaaS — source onboarding UI, backend, Kafka-driven profiling, LLM enrichment, steward approval, REST + MCP exposure.
  • Tool Registry — connection pools, per-source-type tools, sync + async API execution, Vault integration, audit, governance.
  • Shared platform-contracts library (proto + generated Java stubs).
  • Operability — observability, deployment, capacity planning for the above.

3.2 Out of scope External

  • SKAN.ai capture and the Ingestion Pipeline (upstream producer of SKILL.md).
  • SKILL.md authoring and Python placeholder generation.
  • Google ADK agent runtime (downstream consumer).
  • Business glossary curation tool (we import; we do not author).

3.3 Assumptions

  • HashiCorp Vault (or an enterprise-approved equivalent) is available for secret storage.
  • Kafka cluster is available for event streaming.
  • Postgres is approved for service backing stores.
  • Kubernetes is the deployment target.
  • An enterprise LLM endpoint is available (Claude, internal gateway, or equivalent).

4.Business Context — O2A

Operations teams across business lines execute Standard Operating Procedures that validate records across structured databases, REST APIs, and unstructured documents in S3. The work is time-consuming, repetitive, error-prone, and unauditable when distributed across humans. O2A captures the work via SKAN.ai, codifies it as SKILL.md, generates agents that perform the secondary validation autonomously, and runs them at scale.

The decisive bottleneck for accuracy is data-awareness: agents fail when they don't know which table holds the record, what a column means, which API operation answers the question, or how two datasets relate. DDaaS solves discovery; Tool Registry solves governed execution. Together they make automation scalable rather than artisanal.

6.Component 1 — Data Dictionary as a Service

6.1 Responsibilities

  • Collect source connection details via UI and REST API.
  • Persist source metadata; write credentials to Vault.
  • Trigger asynchronous profiling via Kafka.
  • Profile each source object: DB tables (first 100 rows), S3 prefixes (sampled), API operations (from OpenAPI spec), Mongo collections.
  • Use an LLM to enrich profiles with column semantics, glossary mappings, and cross-object relationships.
  • Run a steward approval workflow.
  • Expose the approved dictionary as REST and MCP — searchable, describable, consumable by the Ingestion Pipeline.

6.2 Source registration flow

User ─▶ DDaaS UI ─▶ POST /v1/sources
                       │
                       ├─▶ validate payload
                       ├─▶ Vault.write(credentials) → vault_ref
                       ├─▶ Postgres.insert(source)
                       ├─▶ Kafka.publish(ddaas.source.registered, {source_id})
                       └─▶ 202 Accepted { source_id, status: "profiling" }

6.3 Profiling workers

One pool of Spring Boot consumers subscribed to ddaas.source.registered. A factory selects the right profiler strategy per source type. Profilers call the Tool Registry for connection access — DDaaS does not open driver connections itself, preserving the single execution choke point.

Source typeStrategyNotes
Oracle / MSSQL / Teradata / Postgres / MySQLSELECT with dialect-aware FETCH FIRST 100 ROWS ONLY / LIMIT 100 / TOP 100 / SAMPLE 100 per column metadataProfiler also pulls column types, nullability, distinct-in-sample, min/max, top-K values via aggregations on the sample.
S3Detect format → parquet metadata or CSV header → read first 100 rows of first matching fileFor parquet, schema is read from footer; no full scan.
REST API (OpenAPI)Parse spec → emit one virtual object per operationNo live calls during profiling. Request/response schemas become the dictionary entry.
MongoDBfind({}).limit(100) per collectionSchema inferred from sampled documents; nested types preserved.

6.5 LLM enrichment

A dedicated worker consumes ddaas.profile.completed, batches columns per object, and calls the LLM with a strict, schema-validated prompt. The LLM produces:

  • Column description — plain-English semantic meaning derived from name + sample values.
  • Glossary term mapping — best matching corporate glossary entry, with confidence.
  • Relationship candidates — across previously profiled columns, ranked by name similarity + value-overlap + LLM judgment.

Output is written as dictionary_entry rows in pending_approval. Failures fall back to a stub description and flag the column for manual authoring.

6.6 Approval workflow

  • Single approve — steward reviews per column in UI.
  • Bulk approve with rules — e.g. auto-approve all columns with confidence ≥ 0.9 and no PII flag.
  • Override — steward edits description before approval; the LLM's output is preserved as an audit trail.

Only approved entries are exposed via REST / MCP.

7.Component 2 — Tool Registry

7.1 Responsibilities

  • Be the only component that opens connections to data sources.
  • Provide a Tool implementation per source type, behind a uniform interface.
  • Manage pooled connections, credentials, and source-side governance.
  • Execute sync and async API operations with a uniform {trace_id, status, result} response shape.
  • Emit a complete audit record for every call.
No-LLM constraint Tool Registry must not import any LLM SDK. All semantic work has already happened upstream in DDaaS. This keeps the execution path predictable, low-latency, and easy to security-review.

7.2 SourceTool SPI

public interface SourceTool {
  String type();                                          // "oracle", "s3", "api-async", ...
  HealthStatus testConnection(SourceSpec spec);
  ObjectSchema describeSchema(SourceRef src, ObjectRef o);
  SampleResult sampleRows(SourceRef src, ObjectRef o, int n);
  ExecutionResult execute(ExecutionRequest req);          // sync
  AsyncHandle   executeAsync(ExecutionRequest req);       // async
}

Each SourceTool is a Spring Boot starter module. Registering a new source type means adding a module — no core changes, no rebuild of the router.

7.4 Sync API tool

Agent ─▶ ToolRegistry.execute(tool_id=apisync://kyc.verify, payload)
            │
            ├─▶ resolve operation from OpenAPI
            ├─▶ Vault.read(credentials)  (cached per pool lifecycle)
            ├─▶ WebClient.post(URL, headers, body).timeout(T)
            ├─▶ audit + metrics
            └─▶ response { status, body } back to agent

7.5 Async API tool

Agent ─▶ ToolRegistry.executeAsync(tool_id=apiasync://onboard.start, payload)
            │
            ├─▶ submit upstream request → upstream returns request_id
            ├─▶ create trace { trace_id, request_id, status: "pending" }
            ├─▶ MODE A: webhook
            │     upstream → ToolRegistry /callbacks/{trace_id}
            │     → trace.status = "complete", store result
            ├─▶ MODE B: polling
            │     scheduler polls upstream status endpoint with backoff
            │     → on success, trace.status = "complete"
            └─▶ agent calls GET /v1/traces/{trace_id} to retrieve result
                (or subscribes to a server-sent-event stream)

7.6 Execute pipeline (all tools)

  1. Authenticate caller (mTLS + signed header carrying agent identity).
  2. Resolve tool_id → SourceTool + source_id.
  3. Check per-caller and per-source quota / rate limit.
  4. Acquire connection from pool (lazy pool init on first call).
  5. Apply timeout, row cap, byte cap.
  6. Execute parameterized statement / call API.
  7. Stream result; chunk by configured page size.
  8. Emit audit event; update Resilience4j circuit breaker.

8.Connection Pool Strategy (HikariCP-independent)

Design decision The JDBC pool implementation is a configuration choice per source, not an architectural commitment. If HikariCP is not on the approved stack, Tool Registry runs unchanged on Oracle UCP, Apache DBCP2, or Tomcat JDBC Pool.

8.1 ConnectionPoolProvider SPI

public interface ConnectionPoolProvider {
  String id();                                            // "hikari" | "ucp" | "dbcp2" | "tomcat"
  PoolHandle initialize(SourceSpec spec, Credentials c, PoolTuning t);
  Connection acquire(PoolHandle h);
  void release(PoolHandle h, Connection c);
  PoolMetrics metrics(PoolHandle h);
  void rotateCredentials(PoolHandle h, Credentials newC);
  void shutdown(PoolHandle h);
}

Each provider is a Spring Boot starter; only one is on the classpath in any given deployment, selected by configuration.

8.2 Provider comparison

ProviderStrengthWeaknessBest for
Oracle UCPRAC-aware, FAN/FCF, sharding support, Oracle-blessedTied to Oracle stack; verbose configOracle-heavy enterprises Default for OracleTool
HikariCPFastest, smallest footprint, simple configMay not be on conservative approval listsGreenfield or progressive shops
Apache DBCP2Apache Software Foundation, broadly approved, matureSlower than Hikari/UCP under heavy contentionConservative enterprises Universal fallback
Tomcat JDBC PoolShips with Tomcat; lightweightLess features than UCP; tuning requiredShops standardizing on Tomcat

8.3 Configuration example

tool-registry:
  pools:
    default-provider: dbcp2          # org-wide approved default
    overrides:
      oracle:                        # every Oracle source
        provider: ucp
        tuning: { min: 5, max: 60, idle-timeout-ms: 60000, max-lifetime-ms: 1800000 }
      postgres:
        provider: dbcp2
        tuning: { min: 5, max: 80, idle-timeout-ms: 30000, max-lifetime-ms: 1200000 }
      teradata:
        provider: dbcp2
        tuning: { min: 2, max: 20, idle-timeout-ms: 90000, max-lifetime-ms: 1800000 }

8.4 Non-JDBC pooling

Mongo, S3, and API tools do not go through this SPI — they use native pools provided by their drivers (Mongo driver pool, AWS SDK connection pool, Reactor Netty pool). This is correct: the SPI exists to insulate JDBC from a single pool product.

8.5 Operational properties guaranteed across providers

  • Lazy initialization on first call; warmable on demand.
  • Credentials never logged.
  • Credential rotation triggers a controlled pool drain + refresh.
  • Metrics exposed via Micrometer with provider-agnostic names: pool.active, pool.idle, pool.wait_ms, pool.acquire_failures.

9.Integration Contracts

9.1 With the Ingestion Pipeline (upstream consumer of DDaaS)

The Pipeline calls DDaaS at SKILL.md build time:

1. Pipeline → POST /v1/tools/search { "q": "customer outstanding balance", "k": 8 }
   ◀── ranked Tool IDs + summaries
2. Pipeline → GET  /v1/tools/{id}/describe
   ◀── full dictionary entry incl. execution_hint
3. Pipeline embeds tool_id + describe excerpt into SKILL.md and (if generated)
   into the Python placeholder for ADK runtime.

9.2 With Google ADK (downstream consumer of Tool Registry)

The generated agent calls Tool Registry at runtime:

1. ADK Agent ─▶ ToolRegistry.execute({ tool_id, params })
2. Tool Registry ─▶ source ─▶ result
3. Tool Registry ─▶ Audit log
4. Agent continues SOP using result

9.3 Internal DDaaS ↔ Tool Registry

DDaaS profilers call Tool Registry for sampling. This keeps a single execution choke point.

DDaaS Profiler ─▶ ToolRegistry.sampleRows(source_id, object_id, 100)
                ◀── rows + native schema
DDaaS persists column_profile rows; emits ddaas.profile.completed.

11.Cross-Cutting Concerns

11.1 Security

  • Credentials live in Vault only. Both services hold only Vault references. Only Tool Registry resolves them, at pool init.
  • mTLS for service-to-service traffic; caller identity carried in mTLS cert + signed header.
  • RBAC on DDaaS UI: Source Owner, Data Steward, Platform Admin, Read-Only Viewer.
  • Statement templates only — no raw SQL passthrough from external callers.
  • Read-only by default in v1; write-back Tools are deferred and separately governed.

11.2 Observability

  • OpenTelemetry tracing end-to-end: Ingestion Pipeline → DDaaS → Tool Registry → source.
  • Micrometer metrics with provider-agnostic names; Prometheus scrape.
  • Structured JSON logging; correlation IDs propagate via gRPC metadata.
  • Pre-built Grafana dashboards: pool saturation, query latency, profiling lag, LLM enrichment lag, audit rate.

11.3 Resilience

  • Resilience4j circuit breaker per source; degraded sources marked in DDaaS UI.
  • Per-tool retry policy — idempotent reads only.
  • Kafka consumer DLQ for poison events; retries with exponential backoff.
  • Pool acquire timeout shorter than query timeout to surface saturation early.

11.4 Configuration management

  • Spring Cloud Config (or equivalent) for environment-specific overrides.
  • Per-source pool tuning is hot-reloadable without restart.

Pool sizing rule of thumb: per source, max = min(source_supports,  ceil(peak_qps × avg_latency_s × 1.3)), total cluster-wide < source's configured connection limit divided by replica count.

16.Risks & Mitigations

17.Open Decisions

  1. Approved JDBC pool — confirm enterprise stance (UCP for Oracle, DBCP2 elsewhere is the safest default).
  2. Vault flavor — HashiCorp Vault or AWS Secrets Manager?
  3. Profiling inline vs always-async — recommend always-async with UI status badge.
  4. Glossary source — import from existing tool or author in-platform?
  5. Sampling for very large tables — first 100 (v1) vs stratified per partition (v2).
  6. Caller identity — per-agent service account vs shared ADK identity.
  7. Async API uniformity — webhook-first with polling fallback, or always-poll for uniformity?
  8. Write-back tools — confirmed out of v1?
  9. Tool ID stability across schema changes — stable ID, schema versioned underneath (recommended).
