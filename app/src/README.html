Internal Platform Initiative

Data-Aware Agent Platform

A two-service foundation — Tool Registry and Data Gateway — that turns data discovery, integration, and governed execution into platform capabilities so that LLM agents generated from SKILLS or SOPs are data-aware by construction and run at enterprise scale.

1.Executive Overview

Enterprises are trying to automate Standard Operating Procedures with LLM agents, but the agents keep failing because they don't know the data. Schemas live in tribal knowledge, credentials are scattered, every agent reimplements connectivity, and no team can answer "what data exists and who is allowed to read it" with a single source of truth.

This platform fixes the structural cause, not the symptom. It introduces two independently deployable services:

Tool Registry

The system of record for meaning. Catalogs every data source, profiles every object, uses an LLM to produce a curated Data Dictionary (semantics, glossary mapping, join-key candidates), and exposes each approved object as a callable Tool over MCP and REST.

Data Gateway

The system of record for access. Owns pooled connections to every registered source, resolves credentials from Vault, executes parameterized queries within enforced quotas and timeouts, streams results, and writes a complete audit trail.

The existing Agent Generator is a downstream consumer. It calls the Tool Registry at design time to build data-aware agents, and the generated agents call the Tool Registry's MCP surface at runtime. Neither the generator nor any agent ever touches a database driver or a credential.

2.Problem Statement

2.1 The problem, stated concretely

An SOP says "reconcile the customer's outstanding balance against last quarter's payments." The agent has to guess which database holds customers, which table holds payments, what outstanding maps to, whether cust_id in one table is the same key as customer_ref in another, and what a given date column actually means. Today every team solves this by hand — pasting schemas into prompts, hardcoding SQL into skills, opening a JDBC connection per agent, and rewriting it all when a column gets renamed. The result is automations that are slow to build, brittle in production, impossible to govern, and unable to scale past a handful of SOPs.

2.2 The five structural problems underneath

  1. No catalog of what data exists. Every team rediscovers the same tables. There is no canonical, machine-readable description of any column's meaning, sensitivity, or relationships.
  2. No standard way to expose data to agents. Each agent reimplements connectivity, query templates, and result parsing. Bugs and security gaps multiply per agent.
  3. Credentials are everywhere. DB passwords land in config files, notebooks, and agent prompts. Rotation is a fire drill.
  4. Connections don't scale. When you go from 10 concurrent agents to 1,000, each agent opening its own pool melts the source database.
  5. Governance is impossible. No one can answer "which agents read this table last week?" or "who approved exposing that column?" because there is no choke point.
Shared root cause: Data discovery, data integration, and data execution are tangled into every agent, instead of being platform capabilities.

3.Why Two Services, Not One

These problems split cleanly along a metadata vs. execution line, and that line should become a service boundary. Collapsing both into one service is tempting on day one and painful by month six.

DimensionTool RegistryData Gateway
Primary workloadMetadata reads, semantic search, LLM enrichment (offline), human approvalActive query execution, connection pooling, streaming I/O
Scale signalCatalog size & read QPSConcurrent active queries & connection capacity per source
StateStateless API + Postgres + vector indexStateful in memory: pools, circuit breakers, in-flight queries
ConsumersMany & varied — agents, generator, dashboards, humansExactly one — the Tool Registry
Security postureHolds dictionary, glossary, Tool definitions. No credentials.Only component that holds live DB credentials in memory.
Release cadenceHigh — new tools, dictionary edits, glossary changes ship dailyLow — new connector types, pool tuning, gateway upgrades ship rarely
Blast radius of a bad deployCatalog UI wrong for a few minutesEvery running agent's query fails
External surfaceMCP + REST + UIInternal gRPC only — no MCP, no UI

The separation is not architectural neatness; it is the only way to give each concern the operational treatment it deserves — independent scaling, independent release pipelines, independent failure modes, and a narrowly hardened security perimeter around the credentialed surface.

Bonus reuse: Data Gateway, free of MCP and dictionary concerns, becomes reusable beyond agents — BI tools, data quality jobs, and ad-hoc query consoles can all sit on top of it.

4.How This Solves the Problem at Scale

The architecture turns each of the five structural problems into a one-time platform investment instead of a per-agent tax.

4.1 Discovery becomes asynchronous and cached

Profiling and LLM enrichment run offline, on a schedule, against samples — not on the agent's hot path. The dictionary is read 1,000s of times for every time it's written, and 99% of reads hit a cache. Adding a new SOP doesn't trigger a single new database query.

4.2 Integration becomes a contract, not a copy-paste

Every object in every source is a Tool with the same describe / query / sample shape, exposed over MCP and REST. New agents inherit connectivity for free. When a column is renamed, one dictionary update propagates to every consumer.

4.3 Credentials live in exactly one place

Tool Registry never sees a password. Data Gateway resolves credentials from Vault at pool init and refreshes on rotation. Rotating a DB password becomes a Vault operation — no agent code changes, no redeploys.

4.4 Connections are pooled across the entire fleet

1,000 concurrent agent runs against Postgres share one HikariCP pool of, say, 80 connections — not 1,000 connections. The source database sees stable, bounded load no matter how many agents are running. The gateway autoscales horizontally on active-query count; each replica adds capacity, not connections per source.

4.5 Governance becomes structural

Every query in the company funnels through one audit log with caller identity, statement hash, row count, latency, and outcome. Quotas, rate limits, query timeouts, and circuit breakers are enforced by the gateway — not by trust in every agent author. The steward approval workflow on the registry side means no column is exposed to an LLM without a human (or a bulk-approval policy) signing off.

4.6 The platform compounds

Once a source is registered, every future agent that needs it gets data-awareness for free. Once a connector type is implemented, every future source of that type onboards in minutes. The marginal cost of the next automation drops sharply — which is the working definition of platform leverage.

5.2 Key invariants

  • External consumers never talk to Data Gateway directly. Tool Registry is the only client of Data Gateway.
  • Data Gateway is the only component that opens a connection to any data source.
  • Credentials live in Vault; both services hold only Vault references. Only Data Gateway can resolve them.
  • Every read in the company passes through exactly one audit choke point inside Data Gateway.
  • Adding a new source type = adding a connector plugin. No core changes in either service.

6.Tool Registry — Detailed Requirements

6.1 Purpose

A governed catalog of every enterprise data source, every object inside it, and a curated Data Dictionary per object. Exposes each approved object as a callable Tool over MCP and REST.

6.2 Source onboarding

  • Register a source via UI and REST API. Types in scope: Postgres, Oracle, Teradata, MS SQL, MySQL, MongoDB, S3. Extensible via a SourceType plugin.
  • Capture: name, type, endpoint, port, database/schema/bucket, auth method, scoped object list (tables / collections / S3 prefix + file glob + format).
  • Mongo input is collections. S3 input is bucket + prefix + format (parquet / csv / json).
  • "Test connection" must pass before save — delegated to Data Gateway.
  • Credentials submitted once, written to Vault by Tool Registry; only a vault reference (secret_id) is persisted in the registry DB.
  • Re-registration is idempotent — updating endpoint or scoping objects must not invalidate existing dictionary entries unless the object disappears.

6.3 Profiling & Data Dictionary

  • On registration, on schedule, and on demand, launch a profiling job per object:
    • Sample first N rows (default 100, per-source override).
    • Capture per column: native type, nullability, distinct count in sample, min/max, top-K values, examples.
    • S3: derive schema from headers / parquet metadata; sample N rows from first file matching the prefix.
  • LLM enrichment produces, per column:
    • Plain-English semantic description.
    • Business glossary mapping (per §6.4).
    • Cross-object relationship candidates (name similarity + value-overlap heuristic + LLM confirmation).
  • Persist as versioned Data Dictionary records: {source, object, column, native_type, samples, description, glossary_term, related_to[], version, approval_status}.
  • Stewards review and approve via UI; bulk-approve-with-exceptions for objects at scale.
  • Only approved entries are exposed as Tools.

6.4 Business Glossary

  • Importable from CSV / JSON: {term, definition, synonyms, owner}.
  • Versioned, editable in UI, used as authoritative input to the LLM mapping pass.

6.5 Tool exposure — the headline API surface

Each approved object becomes a Tool with a stable ID and a uniform shape, regardless of source type.

MethodBehaviorBacked by
describe(tool_id)Returns the approved dictionary entry — columns, types, descriptions, glossary mapping, sample values, related Tools.Local cache + Postgres
query(tool_id, params)Parameterized read. SQL templates for RDBMS, find / aggregate for Mongo, S3 select / parquet read for S3.Data Gateway
sample(tool_id, n)Fresh sample fetch — bypasses dictionary samples.Data Gateway
search_tools(q, k)Semantic search over the dictionary. Returns ranked Tool IDs with descriptions.pgvector / OpenSearch

Two transports for the same surface: MCP server (for LLM tool-use) and REST (for non-MCP consumers and debugging).

6.6 Consumers

  • Agent Generator (design time): calls search_tools + describe to build the data-awareness context it injects into generated agents. Out of scope of this project — documented consumer only.
  • Generated agents (runtime): call describe / query over MCP.
  • Humans: REST + UI for debugging and stewardship.

7.Data Gateway — Detailed Requirements

7.1 Purpose

The single execution layer for every registered source. Owns connection pools, credentials, query brokering, governance. Stateless toward callers; stateful toward sources.

7.2 Connector SPI

  • Pluggable SourceConnector interface implemented per source type. Built-ins: JDBC (Postgres, Oracle, Teradata, MS SQL, MySQL), Mongo, S3.
  • Each connector exposes: testConnection, describeSchema, executeQuery, streamResults, samplePreview.
  • New source type = new connector module, no core changes.

7.3 Connection management

  • Per-source connection pools, sized and tuned independently. JDBC: HikariCP. Mongo: native driver pool. S3: native SDK + transfer manager.
  • Pool config (min/max/idle/timeouts) overridable per source.
  • Lazy pool creation on first call; warm pools on demand.
  • Credentials resolved from Vault at pool init; refreshed on rotation events.

7.4 Query brokering

  • API: executeQuery(source_id, statement_template, params, options).
  • Statement templates only — no raw user SQL passthrough from external callers.
  • Hard-enforced safety: query timeout, row cap, byte cap, parameter binding (no string concat), prepared statements where driver supports them.
  • Streaming response for large result sets via gRPC server-streaming.

7.5 Governance

  • Rate limit per source and per caller identity.
  • Quotas (queries/min, rows/day) per caller.
  • mTLS between Tool Registry and Data Gateway; caller identity carried in mTLS cert + signed header.
  • Every call audited: {caller, source, object, statement_hash, params_hash, rows, latency, status}. 1y retention.

7.6 Profiling support

  • Dedicated sampleRows(source_id, object_id, n) endpoint used by Tool Registry profiling jobs.
  • Separate quota bucket so profiling never starves runtime traffic.

7.7 Failure handling

  • Per-connector retry policy (idempotent reads only).
  • Circuit breaker per source — if a source is degraded, fail fast on subsequent calls and notify Tool Registry to mark Tools as degraded.

8.Inter-Service Contract

Versioned internal gRPC API published by Data Gateway, consumed by Tool Registry. Both repos share a small platform-contracts library (proto definitions + generated Java stubs) published as an internal Maven artifact — never duplicated.

RPCUsed byPurpose

DescribeSchema(SourceRef, ObjectRef)RegistrationEnumerate columns / fields of an object.
SampleRows(SourceRef, ObjectRef, n)Profiling jobsReturn first/sample N rows for LLM enrichment.

Versioning rule: The contract evolves additively. Breaking changes require a new RPC name (e.g. ExecuteQueryV2) and a deprecation window of at least one release.

12.Open Decisions

  1. Sampling strategy for very large tables — first N (cheap, biased) vs random sample (better dictionary, full scan) vs stratified per partition. Recommendation: first N for v1, stratified in v2.
  2. Write access — should query ever permit writes (insert/update/delete) or strictly reads in v1? Recommendation: read-only v1; writes as a separately governed Tool class later.
  3. Glossary source — import from an existing tool (Collibra / Alation / sheet) or author in-platform?
  4. Vault flavor — HashiCorp Vault, AWS Secrets Manager, or internal KMS?
  5. Caller identity model — service-account-per-agent vs shared platform identity. Affects per-caller quotas.
  6. Federated queries — must Data Gateway ever join across sources, or is that always the agent's job? Recommendation: push out of v1.
  7. Tool ID stability — when an object is re-profiled and schema changes, does the Tool ID stay the same? Recommendation: stable Tool ID, schema versioned underneath.

13.Phasing & Delivery Plan

Phase 1 — MVP In scope

  • Two repos scaffolded with shared platform-contracts artifact.
  • Connectors: Postgres, Oracle, S3.
  • Registration UI, "test connection" via gateway, Vault integration.
  • Profiling jobs, LLM enrichment (column semantics + glossary mapping + relationship candidates), steward approval.
  • MCP + REST Tool exposure with describe / query / sample / search_tools.
  • Data Gateway with HikariCP pools, audit log, circuit breakers, OpenTelemetry.

Phase 2 — Coverage & UX Next

  • Add Teradata, MS SQL, MySQL, MongoDB connectors.
  • Glossary import + management UI.
  • Relationship inference UI with confidence scoring.
  • Bulk approval workflow, semantic search at scale.

Phase 3 — Enterprise hardening Later

  • Stratified sampling for large tables.
  • Write-back Tools as a separately governed class.
  • Federated queries (cross-source joins inside the gateway).
  • Multi-region gateway, active-active with regional Vault.
  • PII / sensitivity auto-classification.
