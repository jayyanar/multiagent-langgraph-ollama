Data Dictionary as a Service (DDaaS) — Architecture & Flow

How ddaas-api, ddaas-profiler, and ddaas-enricher cooperate over Kafka to discover enterprise data, build an LLM-enriched catalog, and expose it to downstream consumers — shown end-to-end with the Tool Registry context.

API service (stateless, HTTP) Worker service (Kafka-driven) Kafka topic Datastore / index Vault LLM endpoint External system

End-to-end flow (the numbered steps)

1
Register source. Source Owner posts DB / API / S3 / Mongo details to ddaas-api.
2
Secure & persist. ddaas-api writes credentials to Vault, persists the source row in Postgres with only a vault_ref.
3
Emit registration event. ddaas-api publishes to ddaas.source.registered.
4
Profiler consumes. ddaas-profiler picks up the event, selects the right strategy by source type.
5
Sample via Tool Registry. Profiler calls sampleRows() on Tool Registry — DDaaS never opens a driver itself.
6
Emit profile. Column types, stats, top-K, samples written to Postgres; ddaas.profile.completed published.
7
Enricher consumes. ddaas-enricher reads profile, batches columns per object.
8
LLM enrichment. Descriptions, glossary mapping, relationship candidates — written as pending_approval.
9
Steward approval. Single, bulk-by-rule, or override; only approved entries reach the catalog.
10
Catalog consumed. Ingestion Pipeline (at SKILL.md build) and ADK (at runtime) call describe / search over REST or MCP.

Service responsibilities

API ddaas-api

  • Hosts the registration UI server, REST API, and MCP surface.
  • Owns the secret-write path to Vault.
  • Serves the catalog read path with cache + pgvector.
  • Runs the steward approval endpoints.
  • Stateless. HPA on RPS.

Worker ddaas-profiler

  • Kafka consumer on ddaas.source.registered.
  • One strategy per source type behind a factory.
  • Calls Tool Registry — does not open connections.
  • Writes column_profile; emits profile.completed.
  • HPA on Kafka consumer lag.

Worker ddaas-enricher

  • Kafka consumer on ddaas.profile.completed.
  • Only DDaaS component that uses the LLM.
  • Concurrency-capped to control cost.
  • Writes dictionary_entry rows in pending.
  • HPA on lag + LLM rate budget.

Why three services, not one?

Different scale signals (RPS vs Kafka lag vs LLM-bound), different failure modes, different blast radius. Independent deploy, independent autoscale, independent on-call.

Kafka topic contracts (DDaaS owns)

  • ddaas.source.registered — emitted by ddaas-api on every successful source POST. Consumed by ddaas-profiler.
  • ddaas.profile.completed — emitted by ddaas-profiler after sampling + stats. Consumed by ddaas-enricher.
  • ddaas.dictionary.updated — emitted by ddaas-api on steward approval. Consumed by catalog cache invalidator and (optionally) the Ingestion Pipeline.
